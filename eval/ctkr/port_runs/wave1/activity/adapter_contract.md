# Proposed adapter contract — activity log

> Generated by `ctkr propose-adapter` from pipeline artifacts + scoped source.
> Mutators append events to the log; projections read over it, as-of a timestamp.

```typescript
export interface ActivityLogAdapter {
  recordActivity(input: { name: string; occurredAt: Instant; status: 'pending' | 'done'; assetHandles: readonly AssetHandle[]; notes?: string }): Promise<ActivityLogHandle>;
  listActivityLogsForAsset(asset: AssetHandle, asOf: HlcTimestamp, options?: { status?: 'pending' | 'done'; limit?: number }): Promise<readonly ActivityLogView[]>;
  getFirstActivityLogForAsset(asset: AssetHandle, asOf: HlcTimestamp, status?: 'pending' | 'done'): Promise<ActivityLogView | null>;
}

export function makeAdapter(): ActivityLogAdapter;
```

## Methods

| method | kind | as-of | semantics | derived_from |
|---|---|---|---|---|
| `recordActivity` | mutator | no | Appends one activity-log event, linked to all supplied assets, and returns its collision-free replica-scoped client id (prefix_replicaId~counter); the activity is a log event (isLog:true). | Activity; AssetAddLog::create; AssetAddLog::executeMultiple |
| `listActivityLogsForAsset` | projection | yes | Returns activity-log views linked to asset and visible at-or-before asOf, optionally filtered by status, ordered newest occurredAt first with HLC as deterministic tie-breaker; limit truncates that order. | AssetLogsInterface::getLogs; AssetLogs::getLogs; AssetLogs::query; LogQueryFactoryInterface::getQuery |
| `getFirstActivityLogForAsset` | projection | yes | Returns the first (newest under the activity-log ordering) matching activity log linked to asset at-or-before asOf, or null when none exists. | AssetLogsInterface::getFirstLog; AssetLogs::getFirstLog |

## Rationale

The sole state-changing operation is recording an activity as an immutable domain event; associating multiple selected assets is atomic within that one event, matching the add-log action without a central multi-write transaction. ActivityLogHandle is an opaque replica-scoped client id using the bound rule `prefix_replicaId~counter`. Both asset-log reads are materialized, time-bounded projections because later appended activity events can change their result. `getFirstActivityLogForAsset` is retained as the explicit source-level first-log semantic rather than requiring callers to infer it from list ordering. No quantity, movement, membership, or access surface is included because this feature inventory supplies no activity-log semantic requiring it.
