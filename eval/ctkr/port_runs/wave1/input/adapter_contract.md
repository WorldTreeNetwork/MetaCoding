# Proposed adapter contract — input log

> Generated by `ctkr propose-adapter` from pipeline artifacts + scoped source.
> Mutators append events to the log; projections read over it, as-of a timestamp.

```typescript
export interface InputLogAdapter {
  recordMaterialInput(occurredAt: Date, materialQuantities: ReadonlyArray<MaterialQuantity>, initialStatus: LogStatus): Promise<InputLogHandle>;
  listInputLogs(filter: { materialType?: MaterialTypeHandle }, asOf: Date): Promise<ReadonlyArray<InputLogView>>;
}

export function makeInputLogAdapter(): InputLogAdapter;
```

## Methods

| method | kind | as-of | semantics | derived_from |
|---|---|---|---|---|
| `recordMaterialInput` | mutator | no | Appends one InputRecorded event for a non-empty set of material-backed quantities and returns its input-log handle; the created handle uses replica-scoped client id (prefix_replicaId~counter); collision-free by construction, no merge step needed. | Input::buildFieldDefinitions; Material::buildFieldDefinitions |
| `listInputLogs` | projection | yes | Returns input-log views materialized from events visible at-or-before asOf, optionally retaining only logs having at least one material quantity whose material has the requested material type; each result exposes its recorded quantities and status as of that time. | LogQuantityMaterialType::query; LogQuantityMaterialType::buildExtraOptionsForm; LogQuantityMaterialType::defineOptions; ViewsExecutionHooks::viewsPreView; ViewsHooks::viewsData |

## Rationale

The source-specific behavior is a material-backed quantity field on input logs plus a Views material-type filter, so the surface has one domain event and one filtered materialized-view read. Drupal Views form/options/data-hook plumbing is collapsed into the typed materialType filter rather than exposed as UI APIs. The query is as-of-time parameterized because both newly appended input events and later log/material state events can change the projected result.
