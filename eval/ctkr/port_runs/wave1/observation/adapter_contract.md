# Proposed adapter contract — observation

> Generated by `ctkr propose-adapter` from pipeline artifacts + scoped source.
> Mutators append events to the log; projections read over it, as-of a timestamp.

```typescript
export interface ObservationAdapter {
  startObservationAddSelection(actor: ActorHandle, selectedAssetIds: AssetHandle[]): Promise<PendingObservationSelectionHandle>;
  confirmObservationAddSelection(actor: ActorHandle, selection: PendingObservationSelectionHandle): Promise<ObservationAssetQueryHandle>;
  recordObservation(assetIds: AssetHandle[], quantityRevisions: QuantityRevisionHandle[], status: ObservationStatus, occurredAt: Date, notes?: string | undefined): Promise<ObservationHandle>;
  cloneObservation(source: ObservationHandle, occurredAt: Date): Promise<ObservationHandle>;
  recordQuantityDeletion(quantity: QuantityHandle, occurredAt: Date): Promise<QuantityDeletionHandle>;
  recordObservationDeletion(observation: ObservationHandle, occurredAt: Date): Promise<ObservationDeletionHandle>;
  getObservationAssetPrepopulation(operation: 'create' | 'edit', rawAssetQueryIds: AssetHandle[], asOf: Date): Promise<AssetHandle[]>;
  getConfirmedObservationAssetQuery(assetQuery: ObservationAssetQueryHandle, asOf: Date): Promise<AssetHandle[]>;
  getObservation(observation: ObservationHandle, asOf: Date): Promise<ObservationSnapshot | null>;
  listObservationRevisions(observation: ObservationHandle, asOf: Date): Promise<ObservationRevisionSnapshot[]>;
  listObservationsForAsset(asset: AssetHandle, asOf: Date): Promise<ObservationHandle[]>;
  getFirstObservationForAsset(asset: AssetHandle, asOf: Date): Promise<ObservationHandle | null>;
  getDefaultQuantityType(asOf: Date): Promise<QuantityTypeHandle | null>;
}

export function makeObservationAdapter(): ObservationAdapter;
```

## Methods

| method | kind | as-of | semantics | derived_from |
|---|---|---|---|---|
| `startObservationAddSelection` | mutator | no | Records a user-scoped pending add-observation selection from the assets selectable by the actor's locally observed update-access decision. | AssetAddLog::executeMultiple |
| `confirmObservationAddSelection` | mutator | no | Consumes the pending selection exactly once and creates an asset-query payload containing only assets visible to the actor at confirmation; inaccessible selected assets are silently omitted even when no redirect payload can be produced. | AssetAddLogActionForm::submitForm |
| `recordObservation` | mutator | no | Appends an observation whose asset field is exactly the supplied raw asset ID list, without replacing it with a loaded, viewable-only subset, and whose quantity field may contain zero or more revision-aware quantity references. | AssetAddLog::create |
| `cloneObservation` | mutator | no | Appends a cloned observation and a distinct cloned quantity for every quantity revision referenced by the source, so the clone references new quantity entities rather than the source quantities. | LogQuantityEventSubscriber::logClone |
| `recordQuantityDeletion` | mutator | no | Records deletion of a quantity and revision updates removing only that quantity from every referencing observation, including otherwise undisclosed observations; each affected observation revision carries a message identifying the deleted quantity UUID. | EntityHooks::quantityDelete |
| `recordObservationDeletion` | mutator | no | Records deletion of the observation together with deletion of every quantity currently referenced by that observation's quantity field. | EntityHooks::logDelete |
| `getObservationAssetPrepopulation` | projection | yes | Returns the raw asset query IDs for a new observation create form and returns an empty list for an edit form, so query parameters never prepopulate an existing observation. | FormHooks::entityPrepareForm |
| `getConfirmedObservationAssetQuery` | projection | yes | Returns the viewable-only asset IDs produced by confirmation, or an empty list once the query payload is no longer available under the as-of log state. | AssetAddLogActionForm::submitForm |
| `getObservation` | projection | yes | Returns the observation state at or before the query time, including its ordered asset references, status, and zero-or-more QuantityRevisionHandle references; returns null when deleted or not disclosed. | invented: a single materialized observation view is required to expose the asset and multi-value revision-aware quantity fields maintained by the entity and field hooks. |
| `listObservationRevisions` | projection | yes | Returns the observation's revision history known at the query time, including quantity-removal revisions and their deleted-quantity UUID messages. | EntityHooks::quantityDelete |
| `listObservationsForAsset` | projection | yes | Returns observations referencing the asset in the log-query ordering at or before the query time, filtered to records disclosed to this replica. | AssetLogsInterface::getLogs |
| `getFirstObservationForAsset` | projection | yes | Returns the first observation for the asset under the same materialized log-query ordering as listObservationsForAsset, or null when none exists. | AssetLogsInterface::getFirstLog |
| `getDefaultQuantityType` | projection | yes | Returns the quantity type selected as the default by the materialized quantity configuration at the query time, or null when no default exists. | FarmLogQuantityHelper::defaultQuantityType |

## Rationale

Writes are represented as domain events: asset-selection confirmation, observation recording and cloning, and observation/quantity deletion. Reads are as-of materialized views because later creation, revision, deletion, configuration, and disclosure events can supersede prior values. Confirmation and prepopulation are separated to preserve both subtle asset behaviors: confirmation silently filters by visibility and consumes its selection, while a newly recorded observation preserves the raw submitted asset list exactly and edit forms never prepopulate from that query. Quantity references are explicitly plural and revision-aware; cloning and both deletion cascades are therefore expressed as event-producing operations, with revision history exposed to verify the required deletion message.
